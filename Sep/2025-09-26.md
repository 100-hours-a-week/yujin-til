### 🚀 Today I Learned

#### 옵티마이저(Optimizer)

- 옵티마이저는 SQL을 **어떻게 실행할지(실행계획)**를 결정하는 DBMS 내부 엔진
- 같은 SQL이라도 데이터 양·분포·인덱스 여부에 따라 실행계획은 달라질 수 있음
- 현재 대부분의 DB는 **CBO(Cost-Based Optimizer)**를 사용하여 통계 정보 기반으로 비용이 가장 낮은 실행계획을 선택

### 🗨️ 오늘의 회고

이번 정리를 통해 **“SQL이 실행되기까지 DB 내부에서 어떤 일이 벌어지는지”**를 구조적으로 이해하게 되었다.
특히 같은 SQL이라도 실행계획은 여러 가지가 될 수 있고,
옵티마이저가 통계 정보와 비용을 기반으로 실행 방법을 선택한다는 점이 인상 깊었다.

이전에는 EXPLAIN 결과를 보더라도
type, Extra, Using filesort 같은 항목들이 단순한 용어처럼 느껴졌다면,
이제는 왜 이런 실행계획이 나왔는지,
그리고 쿼리를 어떻게 바꾸면 옵티마이저의 선택이 달라질지를 고민할 수 있게 되었다.

또한 단순히 “인덱스를 쓰면 빠르다”가 아니라,

- 언제 풀 테이블 스캔이 더 나을 수 있는지
- ORDER BY / GROUP BY에서 인덱스가 어떻게 활용되는지
- 파일소트와 임시 테이블이 왜 성능 병목이 되는지

같은 DB의 기본 데이터 처리 방식을 자세히 이해해볼 수 있었던 의미있는 시간이었던 것 같다.
