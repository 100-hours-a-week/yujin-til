#### spring 제공 예외처리

- 기본 예외처리 : BasicErrorController
  - 예외 발생시, 위 컨트롤러 호출
  - 에러 요청을 다시 was에 전달
  - 클라이언트 입장에선 유용하지 않은 정보
- @REsponseStatus
  - 에러 HTTP상태를 변경하는 어노테이션
  - 단점은 위와 같음
- ResponseStatusException
  - 스프링5에서 추가된 예외 응답
  - HttStatus와 함께 선택적으로 reason, cause 추가
  - Unchecked Exception 상속
  - 한계 : 직접 예외 처리 프로그래밍함(일관되기 어려움)
- **@ExceptionHandler**
  - 매우 유연하게 에러 처리 가능
  - @ResponseStatus와 결합 가능
  - 컨트롤러에서 구현 → 이걸 전역으로 처리하는 기술 제공함(밑에 있는거)
- @ConTrollerAdvice, **@RestControllerAdvice**
  - 스프링 전역으로 익셉션핸들러 적용하도록 자바 빈에 등록됨
  - 두개의 차이는 @ResponseBody 여부
  - 주의 : 프로젝트 당 하나의 ControllerAdvice 관리
- 적용하기 - **“반복을 제거하고 규칙을 강제하기”**
  1. 에러 코드 정의하기 public interface ErrorCode{}해서 implements
  2. 커스텀 에러 코드 정의하기
     - RuntimeException을 상속받는 예외 클래스
     - 서비스 내 발생할 예외를 처리할 클래스 정의
  3. 에러 응답 클래스 정의
  4. @RestControllerAdvice 구현
  5. 에러 응답 확인
  - 번외 - Enum으로 정의

### 🔥 오늘의 도전 과제와 해결 방법

- 스프링 예외 처리 방식을 일원화하기 위해 BusinessException, GlobalExceptionHandler, ErrorResponse 구조를 직접 설계했다. 기존의 IllegalArgumentException 같은 에러를 직직접 던지는 부분을 모두 제거하고, 모든 예외를 전역 핸들러에서 통합 처리하도록 변경했다. 결과적으로 서비스 코드가 단순해지고, 프론트엔드에서도 일관된 JSON 에러 응답을 받을 수 있게 되었다.
