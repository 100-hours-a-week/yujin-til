### 🚀 Today I Learned

#### DOM 렌더링 구조를 컴포넌트 기반으로 분리하는 흐름

- 게시물 리스트를 서버에서 받아온 뒤 forEach로 `<li>`를 만드는 방식에서, 공통 포맷팅 함수(좋아요수·댓글수·조회수)를 따로 빼서 재사용성을 높였다.

#### 상세 페이지 전환 방식 개선

- 리스트에서 클릭된 게시물 ID를 URL 파라미터로 넘겨 상세에서 다시 조회하는 흐름을 구현했다. "리스트에서 넘겨준 데이터만 쓰는 게 아니라 상세 페이지에서 다시 API를 호출해서 최신 상태를 보여줘야 한다”는 점을 명확히 이해하게 됐다.

#### 권한 기반 UI 컨트롤

- 로그인한 사용자 ID와 게시물 작성자의 ID를 비교해서 수정/삭제 버튼을 노출시키는 방식, 그리고 로직 위치(프론트에서만 제어하면 위험하고 백엔드에서도 검증해야 한다는 점)를 다시 정리했다.

#### 게시물 상세 API와 댓글 API 분리의 필요성 이해

- 초기에 게시물 상세와 댓글을 하나의 API로 불러오도록 만들었지만, 댓글은 페이징·더보기 등 목록 성격이 강한 리소스라 게시물 정보와 갱신·요구 패턴이 다르다는 점을 알게 됐다.
- 사용자 경험을 고려하면 댓글 영역만 따로 갱신되는 경우가 많기 때문에, API를 한 번에 묶기보다 역할에 따라 분리하는 것이 확장성과 유지보수 측면에서 훨씬 낫다는 걸 배웠다. → 그래서 게시물 상세는 단건 조회 API, 댓글은 페이징 기반 목록 조회 API로 나누어 설계했다.

### 🗨️ 오늘의 회고

- 단순히 화면만 뿌리는 게 아니라, “언제 API를 호출해야 하고, 어떤 데이터를 들고 있어야 하고, 어떤 시점에 렌더링해야 하는지”, 이 전체 사이클을 설계하는 능력이 조금 더 성장한 느낌이다. 예상보다 DOM 렌더링과 이벤트 핸들링의 구조화가 훨씬 중요하다는 걸 다시 느꼈고, 다음부터는 더 컴포넌트화된 형태로 재사용할 수 있게 개선해보고 싶다.
