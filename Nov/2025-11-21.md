- 딥다이브 : 도커 이미지가 커지면 발생할 수 있는 문제를 설명하고, 이미지 최적화 전략을 서술하시오.

### 🚀 Today I Learned

- 이미지 크기는 절대적인 기준보다 상대적인 개념이라는 것
  → 네트워크, 레지스트리 위치, 캐시 여부에 따라 실제 영향은 달라짐

- 가장 큰 병목은 컨테이너 실행이 아니라 ‘최초 Pull 시간’

- 최적화는 처음부터 과하게 하는 것보다 ‘기본 원칙 + 필요 시 개선’이 효율적

#### Docker 이미지가 커지면 발생할 수 있는 문제들

- 배포/다운로드 시간 증가
- 스토리지 및 캐시 비용 증가
- 보안 리스크 및 공격 표면 확대
- 시작·스케일링 지연

#### Docker 공식 문서에서 강조하는 최적화 전략들

- 최소 Base Image 사용 (예: alpine, JRE 기반)

- 멀티스테이지 빌드로 빌드 레이어 제거

- .dockerignore로 build context 최소화

- 캐시 활용을 위한 COPY 순서 설계

### 🗨️ 오늘의 회고

- 이번에 도커 기반으로 FE/BE를 직접 빌드하고 배포해보면서, 이미지 최적화 딥다이브에서 정리했던 내용들이 실제 작업에 자연스럽게 연결되었다.
  “이론 → 실전 적용” 흐름을 바로 경험할 수 있었고,
  이미지 최적화라는 주제가 단순히 크기를 줄이는 작업이 아니라
  배포 속도, 스케일링 대응, 보안 안정성까지 연결되는 중요한 설계 요소라는 걸 확실히 느꼈다.
  이번 과제 덕분에 앞으로 Dockerfile을 작성할 때 더 구조적으로, 더 근거 있게 접근할 수 있을 것 같다.
